# Game Logic Agent

ゲームロジックの改修・バグ修正を行うエージェント。

## 担当範囲

- `src/gameLogic.js` — 純粋な判定ロジック（脱落判定、順位計算、ユーティリティ）
- `src/roomService.js` — Firebase上のゲーム状態操作（攻撃実行、次ゲーム、リセット処理）
- `src/constants.js` — ゲーム定数（ボード構成、スロット数、状態定義）

## 作業前に必ず確認するファイル

1. `docs/rule-spec.md` — ゲームルール仕様書。ルール変更時はここも同時に更新すること
2. `docs/app-spec.md` — アプリ仕様書。特にセクション4.7（バトル画面の処理フロー）とセクション5（データ設計）
3. `docs/design-decisions.md` — 過去の設計判断ログ。なぜその仕様になったかの経緯

## 重要な設計判断（変更時は要確認）

### 攻撃判定アーキテクチャ
- クライアント側で全プレイヤーの `slots` を読み取って判定する方式
- `roomService.js` の `executeAttack()` に判定ロジックが集約されている
- この関数が最も複雑。変更時は以下の分岐を全て考慮すること：
  - 他プレイヤーのみヒット
  - 自爆のみ（他プレイヤー無傷）→ 連続攻撃なし
  - 他プレイヤーヒット＋自爆
  - 攻撃者自身が脱落 → 手番終了
  - 勝利確定状態（自分以外全員脱落済み）→ 自爆しても脱落しない
  - 同時脱落（同じeliminatedOrder）
  - ゲーム終了（残り1人）→ 即終了

### 連続攻撃
- 最大2回/手番で固定
- `turnAttackCount` (0 or 1) で管理
- 自爆のみの場合は `anyOtherHit` が false なので連続攻撃にならない

### 脱落判定
- `✕` スロットの `revealed` は常に `false`（脱落判定で除外）
- `isEliminated()` 関数（gameLogic.js）で判定

## テスト観点

ロジック変更時は以下のエッジケースを手動で確認：
1. 2人プレイで相手が脱落 → 即ゲーム終了
2. 自爆のみ → 手番終了（連続攻撃なし）
3. 自爆で自分が脱落 → 連続攻撃なし
4. 最後の1人が自爆 → 脱落しない
5. 1回の攻撃で複数人が同時脱落 → 同じeliminatedOrder
6. 同じ文字が手札に複数 → 全てオープン
7. 連続攻撃2回目でヒット → 脱落判定は実行するが手番終了
